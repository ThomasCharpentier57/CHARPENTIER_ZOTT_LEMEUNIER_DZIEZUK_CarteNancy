Compte Rendu

Notre application répartie utilise une architecture hybride, combinant des technologies front-end et back-end pour fournir une expérience utilisateur intégrée et enrichie.

Front-end

La partie front-end de notre application est développée en JavaScript et utilise la bibliothèque Leaflet pour afficher les données sur une carte interactive. Voici comment elle fonctionne :

Visualisation des données sur la carte :

  • Les stations Vélib sont représentées par des marqueurs bleus. En cliquant sur ces marqueurs, les utilisateurs peuvent voir l'adresse de la station, le nombre de vélos disponibles et le nombre de places de parking libres.
  • Les incidents de circulation sont marqués en orange sur la carte. Les détails comme le type d'incident, la date, la localisation et la description sont accessibles via une popup.
  • Pour ce qui est des établissements d'enseignement supérieur, il n'y avait pas d'api adaptée, donc on a décidé d'afficher quand même les lycées et les collèges de Nancy, représentés par des marqueurs rouges.

Interaction avec les services back-end :

  • Un formulaire est disponible pour réserver une table dans un restaurant en dessous de la carte. Lorsqu'un utilisateur clique sur un marqueur de restaurant, il peut voir les détails du restaurant.
  • L'ajout d'un nouveau restaurant est également possible en cliquant sur la carte. Les détails du restaurant sont ensuite ajoutés à notre base de données.

Back-end

La partie back-end de l'application est développée en Java et utilise les services RMI pour accéder aux données et effectuer des opérations sur la base de données. Voici ce qu'elle propose :

Service RMI Restaurants :

  • Permet de récupérer la liste des restaurants de Nancy depuis notre base de données.
  • Permet d'ajouter de nouveaux restaurants à la base de données.
  • Permet de gérer les réservations dans les restaurants enregistrés.

Service RMI Proxy :

  • Contourne les problèmes de CORS en faisant des requêtes HTTP à des serveurs externes pour récupérer des données bloquées.

Proxy HTTPS :

  • Un serveur HTTPS est mis en place pour gérer nos propres API et les services RMI associés.
  • Gère la communication entre le front-end JavaScript et les services back-end Java via des requêtes HTTP sécurisées.

Schéma d'Architecture

Déroulement de l'Application

Initialisation de l'interface utilisateur :

Le chargement initial des données (stations Vélib, incidents de circulation, établissements d'enseignement supérieur) est effectué dès que l'utilisateur accède à la carte.

Interaction Utilisateur :

Les utilisateurs peuvent interagir avec la carte en cliquant sur les différents marqueurs pour obtenir des informations détaillées.

Lancement du serveur :

Il faut lancer le fichier lancer.sh afin de pouvoir lancer toutes les commandes nécessaires à l'affichage du reste des données.
